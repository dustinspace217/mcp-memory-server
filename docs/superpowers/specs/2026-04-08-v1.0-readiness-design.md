# mcp-memory-server v1.0 Readiness — Design Spec

## Goal

Prepare mcp-memory-server for a v1.0 release — stabilize the API surface, fix known bugs, add features that improve recall quality, and ensure the tool schemas and storage format are ready for a semver stability commitment.

## Context

v0.11.0 shipped vector search (sqlite-vec + @huggingface/transformers) and observation supersede. QA pass with 8 agents identified and fixed critical issues. The remaining work falls into six categories: bug fixes, infrastructure, new features, testing, documentation, and hook improvements.

---

## 1. Bug Fixes (from open GitHub issues)

### 1a. #29 — openNodes IN clause not chunked

`SqliteStore.openNodes()` passes the `names` array directly into `IN (${placeholders})`. If >999 names are passed, SQLite throws `SQLITE_MAX_VARIABLE_NUMBER`. Fix: use the same `CHUNK_SIZE=900` chunking pattern already used in `getConnectedRelations`, `buildEntities`, and other methods.

### 1b. #31 — create_entities returns success when all entities are skipped

When ALL entities are name-collisions, the tool returns `isError: false` with `created: []` and `skipped: [...]`. The MCP response text is raw JSON with no human-readable summary. Fix: when `created.length === 0 && skipped.length > 0`, either set `isError: true` or add a clear text summary like "All N entities already exist" before the JSON.

### 1c. #32 — Double normalization sentinels

Both `index.ts:normalizeProjectId()` and `SqliteStore` independently normalize project IDs. The tool layer converts `undefined`/empty to `undefined`, but the store layer has its own sentinel handling. Fix: centralize in `normalizeProjectId()` and have stores trust that input is already normalized. Remove redundant normalization from stores.

### 1d. #49 — JSONL saveGraph leaks .tmp file on writeFile failure

In `jsonl-store.ts`, the `.tmp` file cleanup only runs on `fs.rename` failure. If `fs.writeFile` throws (disk full), the `.tmp` file is left behind. Fix: wrap in try-finally so `.tmp` is always cleaned up.

## 2. Infrastructure

### 2a. Schema version tracking

Replace `pragma('table_info')` heuristics with a `schema_version` table (single row, single integer). Each migration checks `IF version < N` and bumps after completion. This makes future migrations deterministic instead of fragile column-existence checks. Existing databases get version set based on which columns already exist (detect current state once, then use version going forward).

### 2b. Deprecate JSONL backend

Mark `JsonlStore` as deprecated:
- Add `@deprecated` JSDoc to the class
- Log a warning to stderr on construction: "JSONL backend is deprecated and will be removed in v2.0. Migrate to SQLite with MEMORY_FILE_PATH=memory.db"
- Document in README that SQLite is the recommended backend
- Keep auto-migration (JSONL → SQLite) working
- Do NOT remove JsonlStore code — that's a v2.0 breaking change

## 3. New Features

### 3a. Fix totalCount accuracy in searchNodes

`totalCount` currently reflects LIKE-only matches. When vector search adds supplementary entities, `entities.length > totalCount`. Fix: after vector augmentation, set `totalCount = Math.max(totalCount, entities.length)` for unpaginated results. For paginated results: the CTE count is still useful as "LIKE match count" but totalCount should reflect the full result set. Options:
- (A) Report `totalCount` as LIKE count + unique vector additions for the current page
- (B) Add a separate `likeMatchCount` field and keep `totalCount` as the actual entity count

Recommendation: (A) — adjust `totalCount` upward by the number of vector-supplementary entities added on the current page. This is slightly inaccurate for pagination (doesn't account for vector matches that would appear on other pages) but is always >= the true count, which is the safe direction for "should I fetch more?"

### 3b. Entity timeline tool

New MCP tool `entity_timeline` that returns ALL observations for an entity (active AND superseded) in chronological order. Response includes:
- Entity name, type, project, createdAt, updatedAt
- Observations array sorted by `createdAt` ASC, with each observation including: `content`, `createdAt`, `supersededAt` (empty string if active), `status` ("active" or "superseded")

This makes the supersede feature visible — you can see how facts evolved over time.

Implementation: SQLite query `SELECT * FROM observations WHERE entity_id = ? ORDER BY created_at ASC` (no `superseded_at = ''` filter). JSONL backend: throw "not supported" (consistent with supersede).

### 3c. Temporal relations

Add `superseded_at TEXT NOT NULL DEFAULT ''` to the `relations` table, mirroring the observations pattern:
- Empty string = active relation
- ISO timestamp = retired relation

Schema changes:
- `UNIQUE(from_entity, to_entity, relation_type, superseded_at)` — allows the same relation to exist with multiple time windows
- `created_at TEXT NOT NULL DEFAULT '0000-00-00T00:00:00.000Z'` on relations — when the relation was created
- All existing queries add `WHERE superseded_at = ''` (same pattern as observations)

New MCP tool: `invalidate_relations` — takes array of `{ from, to, relationType }`, sets `superseded_at` to current timestamp. Idempotent (ignores already-invalidated relations).

Update `entity_timeline` to include temporal relations — show when relations were created and invalidated.

Migration: table rebuild (same pattern as observations superseded_at migration).

### 3d. Similarity check on observation writes

When `add_observations` or `supersede_observations` inserts new observations, check for semantically similar existing observations on the same entity using vector search. Return near-matches in the response so the LLM caller can decide whether to supersede, append, or ignore.

Implementation:
- After inserting new observations, if the embedding model is ready, embed each new observation
- Query `vec_observations` for the top-3 nearest neighbors filtered to the same entity (excluding the just-inserted observation itself)
- If cosine similarity > 0.85, include in a `similarExisting` array in the response: `{ content, similarity }`
- This is best-effort — if the model isn't ready or embedding fails, `similarExisting` is omitted (not an empty array)

Output schema change: add optional `similarExisting` array to `add_observations` and `supersede_observations` responses.

### 3e. Graceful shutdown

Add `process.on('SIGTERM')` and `process.on('SIGINT')` handlers:
- Set a `shuttingDown` flag
- If an embedding sweep is running, let the current batch finish but don't start new batches
- Close the database connection cleanly
- Exit with code 0

The `runEmbeddingSweep` already checks `if (!this.db) break` — the shutdown handler sets `this.db = null` (or a separate flag) to signal the sweep to stop.

## 4. Testing

### 4a. Vector search integration tests

Add a test file (`__tests__/vector-integration.test.ts`) that loads the real embedding model and runs end-to-end vector search queries. These tests are slow (~10-20s for model load) so they should be in a separate `describe` block that can be skipped with an env var (`SKIP_VECTOR_INTEGRATION=1`).

Test cases:
1. **Semantic match**: Create entities with observations, search with semantically similar (but textually different) query, verify vector results are returned
2. **LIKE + vector dedup**: Search with a query that matches both LIKE and vector, verify no duplicates
3. **Superseded exclusion**: Create observation, supersede it, verify the old observation doesn't appear in vector results
4. **Pagination with vector augmentation**: Create enough entities for pagination, verify vector supplementary entities appear correctly and totalCount is accurate
5. **Similarity check**: Add an observation, then add a semantically similar one to the same entity, verify `similarExisting` is returned

### 4b. MCP tool handler integration tests (covers #23)

Expand `__tests__/mcp-tools.test.ts` to cover:
- `entity_timeline` tool (new)
- `invalidate_relations` tool (new)
- `supersede_observations` tool (already exists, add handler tests)
- Error responses (missing entity, invalid input)
- `create_entities` skip signal (from #31)

## 6. Hook Improvements

These are client-side changes to `~/.claude/settings.json`, not mcp-memory-server code. They improve memory quality at the extraction layer.

### 6a. SessionEnd hook — add "why" context guidance

Add to the prompt:
- **REASONING**: When saving a decision or preference, include WHY — what alternatives were considered, what trade-offs drove the choice, what context would be needed to revisit the decision later
- **CONTEXT OVER FACTS**: `"user prefers tabs — strong JS/HTML/CSS background, readability priority, Python spaces feel like syntactic obligation"` is worth 10x more than `"user prefers tabs"`
- Examples of good vs. poor observations showing the difference

### 6b. PreCompact hook — same "why" guidance

Mirror the SessionEnd changes. PreCompact is the emergency save — it should capture reasoning context that would otherwise be lost to compaction.

### 6c. PostCompact hook — remove hardcoded test text

Remove the "COMPACTION TEST (2026-04-07)" block and the hardcoded session description. Replace with generic "State what was found and what appears to be missing."

### 6d. SessionEnd hook — add observation quality rules

Add guidance:
- Minimum observation quality: every observation should be understandable in 6 months without the conversation context
- Avoid bare facts without context: "uses Go" → "uses Go — migrated from Python in March 2026 for concurrency requirements and deployment simplicity"
- When saving function signatures, include what the function is FOR, not just its shape
