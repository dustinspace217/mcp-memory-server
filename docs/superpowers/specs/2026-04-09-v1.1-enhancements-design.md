# mcp-memory-server v1.1 Enhancements — Design Spec

## Goal

Add features inspired by the mempalace project (milla-jovovich/mempalace) that improve memory recall quality, observability, and drift resistance across Claude Code compactions and session restarts. All features are additive — no breaking changes to existing tools or schemas.

## Goal Alignment

Every feature in this plan must trace to one or more of the project's three durable goals (defined in `CLAUDE.md` → Project Goals) **and** must pass the anti-bloat gate (would Claude actually use this in the moment?). The table below maps each v1.1 enhancement to the goals it serves, with a one-line "why."

If any feature in this table cannot be defended on its goal trace during implementation — drop it or defer it. The plan is the starting point; the goals are the standard.

| # | Feature | A: drift | B: recall | C: quality | Why this trace |
|---|---|:---:|:---:|:---:|---|
| 1 | Point-in-time queries (`as_of`) | ★ | ★ | | Reconstructs state at time T so Claude doesn't act on facts that have evolved since a decision was made |
| 2 | Importance scoring (1-5) | | | ★ | Lets wake-up summaries surface the high-signal observations first instead of dumping everything |
| 3 | L0/L1 context layers | ★ | | ★ | Context that survives compaction without re-derivation — directly attacks the cold-start problem |
| 4 | Memory type classification | | ★ | | Per-observation who/what/why/how tagging makes faceted recall actually possible |
| 5 | `set_observation_metadata` | ★ | | | Correct importance/layer/type in place without destroying history (preserves Goal B while serving A) |
| 6 | Wake-up summary tool | | | ★ | Start each session warm; the single biggest practical gain for Claude's day-to-day usefulness |
| 7 | Pre-write `check_duplicates` | ★ | | | Prevents additive drift before it lands — semantic dedup at the moment of write, not after |
| 8 | Write audit log | ★ | ★ | | Forensic trail: when did we recall a wrong fact and why? Required to debug drift incidents |
| 9 | Stats / health tool | ★ | | | Visibility into the system's own state — coverage %, storage size, vector readiness |
| 10 | Soft-delete on entities (schema v6) | ★ | ★★ | | The single highest-leverage Goal B fix in the PR — without it, deleting an entity erases its history and `as_of` cannot recall it at any prior moment |
| 11 | Name normalization (Layer 1, hard) | ★ | | | Prevents `dustin-space` vs `dustin/space` vs `dustin-space-site` from coexisting as separate entities at write time — eliminates a known drift source |
| 12 | Eviction policy + four-tier degradation | ★ | ★ | | Designed in this doc but **DEFERRED to a follow-on PR**. The user plans to back up the DB and use it indefinitely across machines, so size pressure WILL eventually bind — captured now so the implementation has a north star when needed |

**Notes on the table:**
- Features 1, 7, 8, 9, 10, 11 cluster under Goal A (drift) — this is the area where v1.0 has the weakest tooling and the highest user-reported pain.
- Features 3 and 6 are the two most direct attacks on the compaction cold-start problem and are the highest-priority items for the practical-use test.
- Feature 2 (importance) is foundational for feature 6 (wake-up summary) — they should land in the same release even though they're separate tasks.
- Feature 4 (memory types) needs care during implementation: the taxonomy must be small enough to be usable. If it grows past ~6 categories, that's a sign we're over-designing and should defer the rest.
- **Features 10 and 11 were added to the v1.1 scope on 2026-04-11** after a Phase 1+2 cross-examination review of Phase B Task 3 (`as_of`). The review surfaced that `as_of` is structurally incomplete without entity-level temporality, and that name dedup needs to be enforced at write time before the database accumulates more drift. Both ship in the same PR as `as_of` — see §10 and §11. Feature 12 was designed in the same review but its implementation is deferred — see §12.

## Context

v1.0.0 shipped with temporal relations, entity timeline, similarity check, schema versioning (v1-v5), graceful shutdown, and 310 tests. A code quality review identified 6 fixes (all applied: similarity formula, KNN scaling, shutdown interface, relation dedup, invalidateRelations return count, comment dedup). This plan covers the next round of features drawn from mempalace's architecture (L0-L3 layered memory, temporal knowledge graph, write-ahead log, importance scoring).

### What mempalace does that we don't (yet)

| Mempalace feature | Our equivalent | Gap |
|---|---|---|
| Point-in-time queries (`as_of`) | `entity_timeline` shows full history | No way to query "state at time T" |
| Importance scoring (1-5 scale) | All observations weighted equally | No prioritization for wake-up summaries |
| L0/L1 context layers | None | No "push" layer that survives compaction automatically |
| Memory type classification (halls) | `entityType` on entities only | No observation-level who/what/why/how tagging |
| Wake-up/summary tool | None | Claude starts cold every session |
| Pre-write duplicate check | Similarity check fires post-write | Can't check before committing |
| Write audit log (WAL) | None | No forensics on what mutated when |
| Stats/health tool | `list_projects` only | No counts, coverage %, storage size |
| Metadata management | `supersede_observations` only | Changing importance/layer/type requires destroying the observation |

## 1. Point-in-Time Queries (`as_of` parameter)

### What

Add an optional `as_of` ISO 8601 timestamp parameter to `read_graph`, `search_nodes`, and `open_nodes`. When provided, the query returns the knowledge graph state as it existed at that moment — entities that were active at `as_of`, with only the observations that were active at `as_of`, and only the relations that were active at `as_of`.

### Why

Compaction summaries often reference "the state when we made decision X." Without `as_of`, Claude can only see current state, which may have evolved since the decision was made. This causes drift — Claude acts on stale reasoning because it can't reconstruct the context that produced a prior decision.

### How — SQLite

`as_of` requires temporal filtering on **all three** record kinds (observations, relations, AND entities), gated by a single per-row helper:

- **Observations**: active at T means `created_at <= T AND (superseded_at = '' OR superseded_at > T)`
- **Relations**: active at T means `created_at <= T AND (superseded_at = '' OR superseded_at > T)`
- **Entities**: active at T means `created_at <= T AND (superseded_at = '' OR superseded_at > T)` — see §10 for the schema v6 migration that adds `superseded_at` to the entities table

A `temporalEntFilter`/`temporalObsFilter`/`temporalRelFilter` helper triple builds the WHERE clause + bound params. All three follow the same shape so callers don't have to think about which to use:

```typescript
private temporalObsFilter(asOf: string | undefined): { clause: string; params: string[] } {
  if (asOf === undefined) return { clause: `superseded_at = ''`, params: [] };
  if (asOf === '') throw new Error("as_of cannot be an empty string — pass undefined for current state or a valid ISO 8601 timestamp");
  return {
    clause: `created_at <= ? AND (superseded_at = '' OR superseded_at > ?)`,
    params: [asOf, asOf],
  };
}
```

Two contract details that matter:

1. **`if (asOf === undefined)`, not `if (!asOf)`.** A naive truthy check conflates `undefined` (no temporal filter) with `''` (an invalid value that the cursor fingerprint also encodes as the empty string). Without the strict check, a caller passing `''` from a UI input gets silent current-state with a cursor that *encodes* an `''` fingerprint — guaranteed to collide on page 2 with any new `as_of` query. Throw on `''` so the contract is three-state: `undefined` / valid ISO / throw.

2. **The helpers must be applied to *every* SQL path that materializes entity rows.** `buildEntities()`, `getConnectedRelations()`, the entity-row query inside `readGraph`, the LIKE path inside `searchNodes`, the KNN/vector path inside `searchNodes`, and `openNodes`. The Phase 1 code review found that the vector search path at `sqlite-store.ts:1624` hardcoded `o.superseded_at = ''` and was the single biggest correctness gap — fixed by routing it through `temporalObsFilter(asOf)`.

### Vector search at `as_of` — two-part fix

Routing the WHERE clause through `temporalObsFilter` is necessary but not sufficient. The Phase 1 review (silent-failure-hunter + adversarial-tester #15) caught a compound bug: `supersedeObservations` at `sqlite-store.ts:1181` *deletes* the `vec_observations` row when an observation is superseded. So even with a correct WHERE clause, there's no historical vector data left to query.

**The fix is two-part:**

- **Part A:** `temporalObsFilter(asOf)` in the vec search WHERE clause + bound params.
- **Part B:** Stop deleting the `vec_observations` row inside `supersedeObservations`. The vec table grows monotonically with history. `syncEmbedding()` keeps writing for newly created/updated observations; only the supersede-time DELETE is removed.

**Cost analysis:** ~1.5 KB per superseded vec row at 384 dims × float32. At the current supersession ratio (~14%, measured against the live DB on 2026-04-11) and projected growth (~65K observations/year), this adds ~14 MB/year to the vec table — proportional to historical recall value, well under any reasonable size cap, and *acceptable* for Goal A correctness. See §12 for how this interacts with the deferred eviction policy.

### How — JSONL

**Not supported. The JSONL backend throws** when `as_of` is provided, matching the existing throw pattern at `jsonl-store.ts:512/522/532` for the other temporal features (`invalidateRelations`, `entityTimeline`).

The earlier "log warning + return current state" plan was a silent-failure trap: a user running queries against JSONL during dev and SQLite in production would get *different answers for the same query*, and the warning lands on stderr where the MCP client never sees it. That directly attacks Goal A. Throwing forces the caller to either upgrade to SQLite or stop asking historical questions — both are honest answers.

The same throw applies to `JsonlStore.deleteEntities` once soft-delete ships in §10 — JSONL callers must not get a hard-delete fallback when SQLite gives them soft-delete, or the two backends drift in a Goal-B-violating way.

### How — MCP tools

Add `as_of` parameter to three tool schemas, with offset normalization disabled to prevent timezone drift:

```typescript
asOf: z.string().datetime({ offset: false }).optional()
  .describe("ISO 8601 UTC timestamp (no offset). Returns graph state as of this moment. " +
            "Re-pass on every page of a paginated query — the cursor fingerprint binds to it " +
            "and InvalidCursorError is thrown on mismatch. Omit for current state.")
```

`{ offset: false }` matters because `2026-04-11T12:00:00+05:00` and `2026-04-11T07:00:00Z` represent the same instant but compare differently as lexicographic strings against UTC-stored values. Forcing UTC at the schema layer eliminates an entire class of off-by-five-hours bugs.

### Pagination interaction

`as_of` queries use the same keyset pagination. The cursor fingerprint includes `as_of` so a cursor from one timestamp can't be reused with a different one — `decodeCursor` throws `InvalidCursorError` on mismatch. This is the **silent-failure firewall**: it prevents page 2 of a historical query from accidentally returning current-state data. Two design notes:

1. **Callers re-pass `as_of` on every page.** This is documented in the tool description (above). The alternative (persisting `as_of` inside the cursor payload) was considered and rejected during Phase 2 cross-examination — silent-failure-hunter pointed out that it removes the firewall and makes silent temporal drift possible. Documented contract + cursor binding wins.

2. **The fingerprint must use a strict `=== undefined` check, not truthy.** See the boxed contract above — the same three-state contract applies to fingerprint construction.

### Boundary semantics

The temporal filter uses a half-open interval `[created_at, superseded_at)`:

- `as_of == observation.createdAt` → observation **included** (`<=` on created)
- `as_of == observation.supersededAt` → observation **excluded** (`>` on superseded, not `>=`)

This is a deliberate choice: at the exact instant a supersession lands, the new value is the truth. Without this, a query at `as_of == T` with both an old observation (superseded at T) and a new observation (created at T) would return both, doubling data at the boundary.

**Adversarial-tester #4 flagged this as a sub-millisecond risk** — at `Date.now()` precision (1 ms), a supersede landing in the same millisecond as a query timestamp is plausible on fast loops. Mitigated by tests that pin both inclusive and exclusive boundaries explicitly. Sub-millisecond ISO precision (microsecond callers) is a separate follow-on — see "Lower-priority deferred items" below.

### Edge cases (with test commitments)

Each of these is a Phase 2-confirmed test gap that lands as a real test in this PR:

- **`as_of` in the future** → same as omitting it (current state). Pin: a test asserting `read_graph(asOf=Date.now()+3600_000)` matches `read_graph()`.
- **`as_of` before any data exists** → empty graph for entities AND relations, via all three tools.
- **Entity exists but all its observations were superseded before `as_of`** → spec says the entity is excluded. *The current implementation does NOT do this*, and the existing Test 3 at `__tests__/knowledge-graph.test.ts:387-402` codifies the wrong behavior with an optional-chained assertion that passes either way. Soft-delete on entities (§10) is the structural fix. The test gets flipped to `expect(graph.entities.find(e => e.name === 'LateEntity')).toBeUndefined()`.
- **Cursor fingerprint rejection across `as_of` values** → cursor from page 1 with `T1` reused with `T2` throws `InvalidCursorError`.
- **`as_of` preserved across paginated round-trip** → 41+ entities, page 1 + page 2 see consistent historical view, disjoint entity sets.
- **Boundary tests** → `as_of == observation.createdAt` (inclusive), `as_of == observation.supersededAt` (exclusive).
- **MCP tool-level Zod validation** → `mcp-tools.test.ts` currently has zero `asOf` coverage. Add one test per tool that an invalid string (`"yesterday"`, `"2026-13-40"`) is rejected and a valid ISO is plumbed through.
- **Vector search at `as_of`** → semantic-only fixture (no LIKE substring overlap), supersede an observation, query at `as_of` before supersede, assert the historical match is recoverable via KNN. Plus: KNN result excluded when parent entity is soft-deleted at `as_of`.

### Lower-priority deferred items

These are real but bounded; tracked as follow-on issues, not blockers:

- **Sub-millisecond timestamp handling** (adversarial-tester #5). Production callers always use `Date.now()` (ms precision), so the risk is bounded to direct API users. Tighten Zod regex in a follow-on.
- **`vectorSearchDegraded` warning field** on `PaginatedKnowledgeGraph` (silent-failure-hunter #5). Useful for callers to distinguish "no semantic matches" from "semantic search broke." Pre-dates `as_of`, changes response shape, deserves its own PR.
- **Unscoped relation query duplication** at `sqlite-store.ts:1463-1468` (code-reviewer). Duplicates `temporalRelFilter` logic that's already encapsulated in `getConnectedRelations`. Drift risk is bounded. Refactor in a follow-on.
- **`deleteObservations` hard-delete asymmetry**. Currently still a hard `DELETE`. With soft-delete on entities (§10), observation-level hard-delete is now inconsistent. Either soft-delete observations too or document the asymmetry.

## 2. Importance Scoring on Observations

### What

Add a `importance` field (REAL, 1.0-5.0, default 3.0) to observations. Higher importance = more likely to surface in wake-up summaries and search results.

### Why

Not all observations are equal. "User prefers tabs" (importance 3) is less critical than "NEVER run sudo in Claude sessions" (importance 5). Without importance, the wake-up tool has no basis for selecting the most critical facts to include in its limited token budget.

### Schema — migration v5→v6

```sql
ALTER TABLE observations ADD COLUMN importance REAL NOT NULL DEFAULT 3.0;
```

Single ALTER TABLE — no table rebuild needed. Default 3.0 means existing observations get medium importance automatically.

Update `schema_version` to 6.

### MCP tool changes

- `create_entities`: observations input gains optional `importance` (default 3.0)
- `add_observations`: each content string can optionally include importance
- `supersede_observations`: replacement observation inherits old observation's importance unless explicitly overridden

To keep the `add_observations` input schema backward-compatible, contents remain `string[]` but a new optional `importances` parallel array is added:

```typescript
observations: z.array(z.object({
  entityName: z.string().min(1).max(500),
  contents: z.array(z.string().min(1).max(5000)).max(100),
  importances: z.array(z.number().min(1).max(5)).max(100).optional()
    .describe("Parallel array of importance scores (1.0-5.0). Omit for default 3.0.")
})).max(100)
```

### Types changes

```typescript
export interface Observation {
  content: string;
  createdAt: string;
  importance: number;  // 1.0-5.0, default 3.0
}
```

### JSONL backend

JSONL serializes importance with default 3.0 on read for legacy data missing the field. No migration needed — JSON round-trips handle missing fields gracefully.

## 3. Wake-Up / Summary Tool

### What

New MCP tool `get_summary` that returns a concise snapshot of the most important and most recent knowledge for a given project (or globally). Designed to be called at session start by Claude Code hooks.

### Why

After compaction or session restart, Claude starts with zero knowledge graph context. The SessionStart hook currently calls `search_nodes`, but this requires knowing what to search for. `get_summary` provides an unprompted overview — the top-N most important facts plus the most recently updated entities.

### Response structure

```typescript
{
  // Top observations by importance, then recency
  topObservations: Array<{
    entityName: string;
    content: string;
    importance: number;
    updatedAt: string;
  }>;
  // Most recently updated entities (last 5)
  recentEntities: Array<{
    name: string;
    entityType: string;
    observationCount: number;
    updatedAt: string;
  }>;
  // Aggregate stats
  stats: {
    totalEntities: number;
    totalObservations: number;
    totalRelations: number;
    projectCount: number;
  };
}
```

### Deduplication with `get_context_layers`

When a SessionStart hook calls both `get_context_layers` and `get_summary`, observations tagged L0/L1 would appear in both responses. To avoid double-loading, `get_summary` accepts an optional `excludeContextLayers` boolean parameter:

```typescript
excludeContextLayers: z.boolean().optional()
  .describe("When true, excludes L0/L1 observations (already covered by get_context_layers). Default false.")
```

When `true`, the topObservations query adds `AND (o.context_layer IS NULL)`, partitioning the observation space: `get_context_layers` owns explicitly layered observations, `get_summary` owns the unlayered rest. Calling `get_summary` alone (without the flag) still returns everything.

### Implementation — SQLite

```sql
-- Top observations by importance DESC, then recency DESC
-- When excludeContextLayers is true, adds: AND o.context_layer IS NULL
SELECT e.name, o.content, o.importance, o.memory_type, e.updated_at
FROM observations o
JOIN entities e ON o.entity_id = e.id
WHERE o.superseded_at = ''
  AND (e.project = ? OR e.project IS NULL)
ORDER BY o.importance DESC, e.updated_at DESC
LIMIT 20;

-- Recent entities
SELECT name, entity_type, updated_at,
  (SELECT COUNT(*) FROM observations WHERE entity_id = e.id AND superseded_at = '') as obs_count
FROM entities e
WHERE (project = ? OR project IS NULL)
ORDER BY updated_at DESC
LIMIT 5;
```

### Token budget

The tool description should note that the response is designed to fit within ~2000 tokens — enough for a SessionStart hook to consume without eating significant context.

### JSONL backend

Supported — sort in-memory by importance (all 3.0 for legacy) then by updatedAt.

## 4. Pre-Write Duplicate Check Tool

### What

New MCP tool `check_duplicates` that accepts candidate observations and returns semantically similar existing observations without writing anything. Allows the caller to decide before committing.

### Why

The current similarity check fires post-write in `add_observations` — the observation is already persisted by the time the caller sees the warning. This tool lets Claude check first, then decide whether to supersede, append, or skip.

### Input schema

```typescript
{
  candidates: z.array(z.object({
    entityName: z.string().min(1).max(500),
    content: z.string().min(1).max(5000)
  })).min(1).max(50)
}
```

### Output

```typescript
{
  results: Array<{
    entityName: string;
    candidateContent: string;
    matches: Array<{
      content: string;
      similarity: number;  // cosine similarity 0.0-1.0
      createdAt: string;
    }>;
  }>;
  modelReady: boolean;  // false if embeddings unavailable
}
```

### Implementation

Reuse the existing KNN infrastructure from `addObservations()`:
1. Embed each candidate with `this.embeddingPipeline.embed()`
2. Query `vec_observations` for k=20 nearest neighbors
3. Filter to same-entity, cosine > 0.80 (slightly lower threshold than the post-write check's 0.85, since this is advisory)
4. Return matches sorted by similarity DESC

### JSONL backend

Returns `modelReady: false` and empty matches (no vector search available). Not an error.

## 5. Write Audit Log

### What

Append-only JSONL file recording every mutation to the knowledge graph. Each line is a JSON object with: timestamp, operation, parameters, and outcome.

### Why

When memory drifts or unexpected state appears, there's currently no way to answer "what happened?" The audit log provides forensics — you can trace how the graph reached its current state without relying on git history of the database file.

### Storage: JSONL file (not SQLite)

Rationale:
- **Append-only** — JSONL is ideal for append-only writes (`fs.appendFile`), no locking needed
- **Isolation** — if the main SQLite DB corrupts or needs migration, the audit trail survives
- **Human-readable** — `cat`, `grep`, `tail -f` work without tooling
- **No migrations** — adding new fields to audit entries never requires ALTER TABLE
- **Rotation** — easy to implement log rotation (rename + compress) independently

### File location

`${MEMORY_FILE_PATH}.audit.jsonl` — sits next to the main database file. For `memory.db`, the audit log is `memory.db.audit.jsonl`.

### Log entry format

```jsonl
{"ts":"2026-04-09T21:30:00.000Z","op":"create_entities","project":"my-project","input":{"count":2,"names":["Foo","Bar"]},"result":{"created":2,"skipped":0}}
{"ts":"2026-04-09T21:30:01.000Z","op":"add_observations","input":{"entityName":"Foo","count":3},"result":{"added":3,"duplicates":0,"similar":1}}
{"ts":"2026-04-09T21:30:02.000Z","op":"supersede_observations","input":{"entityName":"Foo","old":"v1","new":"v2"},"result":"ok"}
```

### What gets logged

Every GraphStore mutation method:
- `createEntities` — names, counts, skipped
- `createRelations` — count, relation summaries
- `addObservations` — entity names, counts, duplicate/similar counts
- `deleteEntities` — names
- `deleteObservations` — entity names, content summaries (truncated to 100 chars)
- `deleteRelations` — count
- `supersedeObservations` — entity names, old→new summaries
- `invalidateRelations` — count, changed count

### What does NOT get logged

- Read operations (`readGraph`, `searchNodes`, `openNodes`, `listProjects`, `entityTimeline`)
- Full observation content (privacy + size — just truncated summaries)
- Embedding operations (too noisy)

### Implementation

Add an `AuditLogger` class in a new `audit.ts` file:

```typescript
export class AuditLogger {
  constructor(private logPath: string) {}
  
  // Appends a single JSON line to the audit log.
  // Uses appendFile (not a write stream) for simplicity — single-writer process.
  async log(op: string, input: Record<string, unknown>, result: Record<string, unknown>): Promise<void> {
    const entry = JSON.stringify({ ts: new Date().toISOString(), op, input, result });
    await fs.appendFile(this.logPath, entry + '\n');
  }
}
```

Audit logging is fire-and-forget from MCP handlers (like embedding sync). Failures are logged to stderr but never fail the operation.

### Configuration

- Enabled by default for SQLite backend
- `MEMORY_AUDIT_LOG=off` env var to disable
- JSONL backend: disabled (the JSONL file itself IS the audit trail, effectively)

## 6. Stats Tool

### What

New MCP tool `graph_stats` that returns aggregate statistics about the knowledge graph.

### Why

`list_projects` returns project names but no counts. There's no way to know how many entities, observations, or relations exist, whether vector coverage is complete, or what schema version the database is running. This information helps Claude (and the user) understand the health of the memory system.

### Response structure

```typescript
{
  schemaVersion: number;
  storage: {
    backend: "sqlite" | "jsonl";
    path: string;
    sizeBytes: number;
  };
  entities: {
    total: number;
    byProject: Array<{ project: string | null; count: number }>;
  };
  observations: {
    total: number;        // active only
    superseded: number;   // retired
  };
  relations: {
    total: number;        // active only
    superseded: number;   // invalidated
  };
  vectorSearch: {
    status: "ready" | "loading" | "failed" | "unavailable" | "off";
    embeddedCount: number;      // observations with embeddings
    coveragePercent: number;    // embeddedCount / total observations * 100
  };
}
```

### Implementation — SQLite

```sql
SELECT COUNT(*) FROM entities;
SELECT project, COUNT(*) FROM entities GROUP BY project;
SELECT COUNT(*) FROM observations WHERE superseded_at = '';
SELECT COUNT(*) FROM observations WHERE superseded_at != '';
SELECT COUNT(*) FROM relations WHERE superseded_at = '';
SELECT COUNT(*) FROM relations WHERE superseded_at != '';
SELECT COUNT(DISTINCT observation_id) FROM vec_observations;
```

Storage size: `fs.stat(dbPath).size`.

### JSONL backend

Supported — compute from in-memory graph. No vector stats (all zeros). Schema version: N/A.

## 7. Context Layers (L0/L1/L2)

### What

Add a `context_layer` field to observations that marks their retrieval tier, inspired by mempalace's L0-L3 layered memory. A new MCP tool `get_context_layers` returns L0+L1 observations in a single token-budgeted response, designed to be called by SessionStart and PostCompact hooks to automatically restore critical context.

### Why — the core drift problem

After compaction or session restart, Claude starts with zero knowledge graph context. The current hooks call `search_nodes`, but that requires knowing what to search for -- a chicken-and-egg problem. The `get_summary` tool (feature #3 above) helps by returning top-importance observations, but it doesn't distinguish between "must always be in context" and "useful if retrieved." That distinction is what causes drift: core rules (like "never run sudo") get treated the same as ephemeral facts (like "current test count is 343"), so critical constraints can fall out of context while stale counts persist.

Mempalace solves this with 4 explicit layers:
- **L0 (~100 tokens)**: Core identity/rules -- always loaded, rarely changes
- **L1 (~500-800 tokens)**: Essential story -- loaded at session start, updated frequently  
- **L2**: On-demand -- retrieved when a topic arises
- **L3**: Deep search -- semantic search for rare queries

We already have L2 (`open_nodes`) and L3 (`search_nodes` + vector search). The gap is L0 and L1 -- the "push" layers that get injected into context without being asked for.

### How — pragmatic implementation

Instead of mempalace's separate L0-L3 files with compression pipelines, we add a single nullable column to observations:

```sql
ALTER TABLE observations ADD COLUMN context_layer TEXT DEFAULT NULL;
-- NULL = L2 (on-demand, the default for all existing observations)
-- 'L0' = always loaded (core identity, rules, constraints)
-- 'L1' = loaded at session start (active work, recent decisions)
```

This is part of the schema v6 migration (alongside `importance`). Existing observations get `NULL` (L2) by default.

### MCP tools

**New tool: `get_context_layers`**

Returns all L0 and L1 observations for a project, sorted by layer (L0 first) then importance DESC.

```typescript
{
  projectId: ProjectIdSchema,
  layers: z.array(z.enum(['L0', 'L1'])).optional()
    .describe("Which layers to return. Defaults to ['L0', 'L1'].")
}
```

Response:

```typescript
{
  L0: Array<{ entityName: string; content: string; importance: number }>;
  L1: Array<{ entityName: string; content: string; importance: number; updatedAt: string }>;
  tokenEstimate: number;  // rough char/4 estimate so hooks can budget
}
```

**Modified tools:**

- `add_observations`: new optional `contextLayers` parallel array (like `importances`)
- `supersede_observations`: replacement inherits old observation's layer unless overridden
- `create_entities`: observations can include layer assignment

### Token budget enforcement

The `get_context_layers` tool enforces a soft token budget:
- L0: max ~100 tokens (~400 chars). If L0 observations exceed this, warn in response.
- L1: max ~800 tokens (~3200 chars). Returns most important first, truncates at budget.

These limits prevent context layers from silently growing until they consume the context window. The response includes `tokenEstimate` so hooks can monitor growth.

### Hook integration

The SessionStart and PostCompact hooks should call `get_context_layers` instead of (or before) `search_nodes`. Example hook change:

```
1. Call get_context_layers for the current project
2. L0 observations are ALWAYS included in the context restoration
3. L1 observations are included if token budget allows
4. Then proceed with search_nodes for project-specific context
```

This is a client-side hook change, not a server change. The server just stores and retrieves the layers.

### Promoting/demoting observations

Observations can be promoted to L0/L1 or demoted to L2 via the `set_observation_metadata` tool (feature #9 below), which updates layer/importance/type in-place without creating a new observation or invalidating its embedding.

### JSONL backend

Store `contextLayer` in the JSON object. `get_context_layers` works by filtering in-memory. No vector search needed.

## 8. Memory Type Classification on Observations

### What

Add a `memory_type` field (TEXT, nullable, free-form with recommended conventions) to observations that classifies the *nature* of the knowledge -- not what entity it's about (that's `entityType`), but what kind of thing was learned.

### Why -- the 5W1H gap

Entity types answer "who/what is this entity?" (person, project, tool). Timestamps answer "when?" Project fields answer "where?" But there's no way to answer "why did we make this decision?" or "how does the user prefer this done?" without keyword-searching observation text. After compaction, Claude can't ask "show me recent decisions about the auth system" -- it can only search for the word "decision" and hope the observation text contains it.

Mempalace classifies memories into 5 halls: decisions, preferences, milestones, problems, emotional. This observation-level classification enables type-filtered queries across entities.

### Recommended values

Free-form string (like `entityType`) with these conventions:

| memory_type | 5W1H dimension | Example observation |
|---|---|---|
| `decision` | Why | "Chose SQLite over Postgres -- no external dependencies needed" |
| `preference` | How | "Always use tabs for indentation" |
| `fact` | What | "Test count is 343" |
| `problem` | What went wrong | "Same-millisecond invalidations collide on UNIQUE constraint" |
| `milestone` | What was achieved | "v1.0.0 shipped with temporal relations and 310 tests" |
| `emotional` | Affective context | "User was frustrated when migration broke twice -- take extra care with schema changes" |

The `emotional` type captures reactions, enthusiasm, frustrations, and interpersonal context. This system isn't only for code recall -- it's for remembering the person: what they care about, what excites them, what to be careful around. Emotional observations add calibration that pure technical facts miss.

### Schema -- part of migration v5 to v6

```sql
ALTER TABLE observations ADD COLUMN memory_type TEXT DEFAULT NULL;
```

Single ALTER TABLE alongside `importance` and `context_layer`. Default NULL means existing observations are unclassified -- callers tag them as they see fit.

### MCP tool changes

- `add_observations`: new optional `memoryTypes` parallel array
- `supersede_observations`: replacement inherits old observation's type unless overridden
- `create_entities`: observations can include type assignment
- `search_nodes`: optional `memoryType` filter parameter

### Filtering in queries

`search_nodes` gains an optional `memoryType` parameter:

```typescript
memoryType: z.string().max(50).optional()
  .describe("Filter to observations of this memory type (e.g., 'decision', 'preference'). Omit for all types.")
```

When provided, entities are included only if they have at least one matching observation. This enables queries like "search for 'auth' but only show decisions."

`get_summary` and `get_context_layers` do NOT filter by type -- they return all observations matching their criteria. Type is metadata for the caller to interpret.

### JSONL backend

Store `memoryType` in the JSON object. Default NULL on read for legacy data. No migration needed.

## 9. Observation Metadata Management (`set_observation_metadata`)

### What

New MCP tool `set_observation_metadata` that updates `importance`, `context_layer`, and/or `memory_type` on existing observations in-place, without superseding (preserving the observation's identity, timestamps, and embedding).

### Why

Without this tool, changing an observation's importance, layer, or type requires `supersede_observations` with the same content -- which creates a new observation row, invalidates the old one, and destroys the existing embedding (requiring re-embedding). That's wasteful for metadata-only changes and produces misleading audit trails ("observation superseded" when nothing about the content changed).

Common operations this enables:
- Promoting an observation from L2 to L0 (making it always-loaded)
- Demoting stale L1 observations back to L2
- Tagging old observations with memory_type after the fact
- Adjusting importance scores as priorities shift

### Input schema

```typescript
{
  entityName: z.string().min(1).max(500),
  content: z.string().min(1).max(5000),  // identifies the observation
  importance: z.number().min(1).max(5).optional(),
  contextLayer: z.enum(['L0', 'L1']).nullable().optional(),
  memoryType: z.string().max(50).nullable().optional()
}
```

At least one of `importance`, `contextLayer`, or `memoryType` must be provided.

### Implementation -- SQLite

```sql
UPDATE observations
SET importance = COALESCE(?, importance),
    context_layer = CASE WHEN ?1 IS NOT NULL THEN ?2 ELSE context_layer END,
    memory_type = CASE WHEN ?3 IS NOT NULL THEN ?4 ELSE memory_type END
WHERE entity_id = (SELECT id FROM entities WHERE name = ?)
  AND content = ?
  AND superseded_at = '';
```

Updates the entity's `updated_at` timestamp. Does NOT touch `vec_observations` -- the embedding is based on content, which hasn't changed.

Returns the updated observation count (0 if entity or observation not found).

### Implementation -- JSONL

Find the observation by content match on the named entity, update fields in-place, save graph.

### GraphStore interface

```typescript
/** Updates metadata (importance, contextLayer, memoryType) on existing active observations.
 *  Does not change content, timestamps, or embeddings. Throws if entity not found. */
setObservationMetadata(updates: SetObservationMetadataInput[]): Promise<number>;
```

## 10. Soft-Delete on Entities (Schema v6→v7)

> **Added 2026-04-11 to v1.1 scope.** Originally planned as a separate Phase later in v1.x, pulled forward after the Phase B Task 3 review made it clear that `as_of` is structurally incomplete without entity-level temporality.

### What

Replace `deleteEntities()`'s hard `DELETE FROM entities WHERE name = ?` with a soft-update that sets a new `superseded_at` column on the entity row, and atomically supersedes all of that entity's observations and invalidates all of its relations at the same timestamp. The entity row stays in the database forever (until eviction — see §12), and historical `as_of` queries can recover it.

### Why — the structural recall gap

Without this, `deleteEntities` violates Goal B in a way that no amount of `as_of` plumbing can fix:

```
T1: createEntities([{ name: 'X', ... }])
T2: addObservations([{ entityName: 'X', contents: ['important fact'] }])
T3: deleteEntities(['X'])
T4: readGraph(asOf=T2)  →  X NOT FOUND, fact unrecoverable
```

Spec line 92 says "entity exists but all its observations were superseded before `as_of` → entity excluded from results." That handles the *information-decay* case, but it doesn't handle the *deletion* case. After a hard delete, there's no row to filter, so even the spec-compliant exclusion logic has nothing to work with. The relation CASCADE makes it worse: every relation involving X also vanishes from history.

This is the **single highest-leverage Goal B fix in the v1.1 PR**. Without it, the memory server cannot faithfully recall when an entity existed in the past after the user issues a delete.

### Why now (not later)

Schema migrations are cheaper the earlier they ship. The v0.10.1 production DB has 132 entities — adding a column and a partial unique index is a sub-second migration. At 132,000 entities (our projected scale at 2 years of usage), the same migration is a multi-minute table rebuild that requires careful planning and a backup. **Migrate now while it's free.**

### Schema — v6 to v7

```sql
-- Add the temporal column. Empty string sentinel = active row.
ALTER TABLE entities ADD COLUMN superseded_at TEXT NOT NULL DEFAULT '';

-- Drop the existing UNIQUE(name) constraint (we can't keep it — historical rows will share names).
-- Replace with a partial unique index that enforces "one active row per name":
CREATE UNIQUE INDEX idx_entities_name_active ON entities(name) WHERE superseded_at = '';

-- (See "Schema mechanics" below for the actual migration approach — SQLite can't drop a column-level
-- UNIQUE constraint without a table rebuild. We use the rebuild pattern from the v4→v5 relations migration.)
```

The partial unique index pattern matches mempalace's approach and is the cleanest alternative we considered. The rejected alternatives were:

| Approach | Why rejected |
|---|---|
| Composite `UNIQUE(name, superseded_at)` | Two soft-deletes of an entity in the same millisecond (plausible under bulk operations) collide on the constraint. Same-timestamp collision is the *exact* failure mode this PR is fixing for relations — repeating it here would be a regression. |
| Composite `UNIQUE(name, created_at)` | Bulk `createEntities` calls (e.g., from SessionEnd hooks) routinely create multiple entities in the same millisecond. Higher collision risk than `superseded_at`, not lower. |
| Incremental ID column + no UNIQUE on name | Works, but every name lookup becomes "find the row with `superseded_at = '' AND name = ?`" — adding overhead to the hot path of the entity lookup. |
| **Partial unique index on `name` WHERE `superseded_at = ''` (chosen)** | Enforces "one active row per name" at the storage layer. Historical rows are unconstrained. Active-row lookups use the index efficiently. Re-creating an entity after soft-delete just works — the new active row claims the name, the old superseded row is invisible to the partial index. |

### Soft-CASCADE — atomic three-way update

Hard delete uses SQLite's `ON DELETE CASCADE` to clean up observations and relations. Soft-delete needs to do the equivalent in application code:

```typescript
// In deleteEntities() — replaces the existing hard-delete logic.
// Wrapped in a single transaction so partial failure rolls everything back.
const now = new Date().toISOString();
const txn = this.db.transaction((names: string[]) => {
  for (const name of names) {
    // 1. Soft-delete the entity itself
    this.db.prepare(`
      UPDATE entities SET superseded_at = ?
      WHERE name = ? AND superseded_at = ''
    `).run(now, name);

    // 2. Supersede all of its active observations at the same instant
    this.db.prepare(`
      UPDATE observations SET superseded_at = ?
      WHERE entity_id = (SELECT id FROM entities WHERE name = ? AND superseded_at = ?)
        AND superseded_at = ''
    `).run(now, name, now);

    // 3. Invalidate all of its active relations at the same instant
    this.db.prepare(`
      UPDATE relations SET superseded_at = ?
      WHERE (from_entity = ? OR to_entity = ?) AND superseded_at = ''
    `).run(now, name, name);
  }
});
txn(names);
```

**Critical:** all three updates use the same `now` value. An `entityTimeline` query at exactly that instant must see a clean cut — either all three sides of the entity are alive, or all three are dead. Any inconsistency here corrupts the history Goal B is supposed to protect.

### Re-create after soft-delete

The partial unique index makes this trivial:

```
T1: createEntities([{ name: 'X', ... }])   → row 1 (active)
T2: deleteEntities(['X'])                    → row 1 (superseded_at = T2)
T3: createEntities([{ name: 'X', ... }])   → row 2 (active, different id)

readGraph()                                  → returns row 2 (current state)
readGraph(asOf=T1.5)                         → returns row 1 (historical state)
readGraph(asOf=T2.5)                         → no active row, no historical row, X excluded
```

Both rows coexist. Queries pick the right one via the temporal filter. The `id` column distinguishes them — relations and observations FK by `id`, not `name`, so historical observations stay attached to the historical entity row.

### Observations and relations FK behavior

Schema v6 leaves the FK constraints in place: observations FK to `entities.id`, relations FK to entity *names* (legacy — the relations table uses name strings, not IDs). This is the existing pattern, not a v6 change.

The soft-delete flow doesn't touch FKs because it doesn't delete rows. The historical observation rows stay attached to the historical (now-superseded) entity row via `entity_id`, and the historical relation rows reference the entity by name + their own `created_at`/`superseded_at` timestamps. `as_of` queries reconstruct the consistent view by JOINing with the temporal filters from §1.

### Goal trace

| Aspect | Goal A (drift) | Goal B (recall) | Goal C (quality) |
|---|:---:|:---:|:---:|
| Eliminates "delete = forget forever" | ✓ | ★★ | |
| Makes Test 3 spec violation moot (entity excluded after soft-delete) | ✓ | | |
| Aligns entity table with relation table's existing temporal pattern | ✓ | ✓ | |

This is the only feature in v1.1 that earns ★★ on Goal B. Every other feature serves recall through better surfacing or filtering — soft-delete is the only one that *prevents data loss*.

## 11. Name Normalization (Layer 1, hard) — and Layer 2 (deferred)

> **Added 2026-04-11 to v1.1 scope.** Driven by a known drift incident: `dustin-space`, `dustin/space`, and `dustin-space-site` ended up coexisting as separate entities for the same project, fragmenting observations across them.

### What

A `normalizeEntityName()` helper applied at the `create_entities` tool boundary in `index.ts`. Two layers, only Layer 1 ships in v1.1:

- **Layer 1 (this PR) — hard normalization, write-time enforcement.** Unicode NFC normalization, case folding, separator stripping. Two names that normalize to the same string are treated as the same entity at create time; the second create either no-ops (existing) or collides (renames).
- **Layer 2 (deferred to follow-on issue) — soft semantic similarity warning.** Embedding-based cosine similarity check at create time. If a candidate name's embedding is within `cosine > 0.85` of an existing name's embedding, return a warning in the create response (not a block). The caller can choose to merge, rename, or proceed.

### Why two layers

Hard normalization catches mechanical drift (whitespace, case, separators). It will not catch semantic drift like `dustin-space` vs `dustinspace217` vs `the personal site`. Semantic similarity catches the second class but introduces false positives that need human judgment. Splitting them lets us ship the cheap-and-correct fix now and defer the harder one.

### Layer 1 — hard normalization

```typescript
// In index.ts, alongside the existing normalizeProjectId helper.
function normalizeEntityName(name: string): string {
  return name
    .normalize('NFC')                    // Unicode NFC: canonical compose. e1+0301 → é.
    .toLowerCase()                       // case fold (Turkish locale concerns? see "Caveats")
    .replace(/[\s\-_/]+/g, '')          // strip whitespace, hyphens, underscores, slashes
    .trim();
}
```

Applied at the MCP tool layer in `create_entities`:

```typescript
// Build a Map<normalized → original> across the input batch + an existing-name lookup.
// If a candidate's normalized form matches an existing entity's normalized form, treat them as the same.
```

**Critical:** Layer 1 is **write-time only**. Existing entity names in the live DB are NOT rewritten — that would be a destructive migration. The helper prevents *future* drift; existing duplicates remain until manually merged (which is what the eventual Layer 2 + a `merge_entities` tool would handle).

### Caveats

- **Turkish dotted/dotless I.** Unicode case folding has a known edge case where `İ`/`i` and `I`/`ı` map differently depending on locale. We use locale-insensitive `.toLowerCase()` and document the edge case. Not a real risk for the current use case (English-only entity names).
- **Stripping `-/_/space` is aggressive** and intentional. It treats `mcp-memory-server` and `mcp memory server` and `mcp_memory_server` as the same name. The cost is that `claude-3` and `claude3` would also collide, which is *desirable* in our context (both refer to the same model).
- **NFC over NFKC.** NFKC would also fold compatibility characters like `①` → `1`, which is over-aggressive for entity names. NFC handles the realistic cases (combining diacritics, presentation forms).

### Layer 2 — semantic similarity warning (DEFERRED)

Captured here so the future implementation has a target:

```typescript
// In create_entities handler, after normalizeEntityName succeeds:
// 1. Embed the candidate name with this.embeddingPipeline.embed(name)
// 2. KNN against vec_entities (a NEW vec0 table mirroring vec_observations but for names)
// 3. If any match has cosine > 0.85, return warning in the create response:
//    { created: [...], skipped: [...], similarNames: [{ candidate, existing, similarity }] }
// 4. Caller decides: merge (via merge_entities, also deferred), rename, or proceed
```

Requires a new `vec_entities` table, an embedding sweep for entity names on first run, and the still-deferred `merge_entities` tool to action the warning. Tracked as a follow-on GitHub issue.

| Aspect | Goal A (drift) | Goal B (recall) | Goal C (quality) |
|---|:---:|:---:|:---:|
| Layer 1 prevents `foo-bar` vs `foo_bar` vs `foobar` divergence at write time | ★ | ✓ | |
| Layer 2 catches semantic divergence (deferred) | ★ | ✓ | |

## 12. Eviction Policy + Four-Tier Degradation (DEFERRED — full design captured for future implementation)

> **Designed 2026-04-11. Implementation deferred to a follow-on PR beyond v1.1.** Captured in full here so the future implementation has an unambiguous target. The user explicitly requested this be designed now: *"I plan to regularly backup the DB and use it indefinitely on future machines/installs. even with 6 yrs runway, we will hit it eventually under planned usage. let's plan to address the eventuality now, even if we dont actually write the PR into v1.1."*

### What

A four-tier degradation chain (Active → Superseded → Tombstoned → Hard-deleted) with **time-allowed, pressure-triggered** eviction. Time grants eligibility; size pressure triggers actual movement. Combined with a 6-month LRU shield and a 1 GB size circuit breaker.

### Why — the size pressure trajectory

At the v0.10.1 production scale (measured 2026-04-11):
- 132 entities, 1,616 observations (1,391 active, 225 superseded — 14% supersession ratio), 153 relations (151 active, 2 superseded)
- 3.8 MB file size after ~9 days of usage

Linear projection forward: ~65K observations/year, ~150 MB/year. **6.5 years to 1 GB cap** at current rates. The user plans to back up the DB and reuse it indefinitely across machines and installs, so this projection isn't theoretical — it's the planned operating mode. Without an eviction policy, we hit the wall.

### Why eligibility-not-trigger

The naive design ("hard-delete anything older than X") fails the user's use case: project breaks of 1-2 years are common, and hard-deleting a project's history during a break would erase exactly the recall the user is preserving the DB for.

The fix: time grants *eligibility*, size pressure *triggers* movement. A 1-year-old superseded observation is eligible to be tombstoned, but it stays in place until the DB approaches the size cap. A 2-year-old tombstoned observation is eligible to be hard-deleted, but it stays in place until pressure forces it out. **No data is lost on a clock alone.** The size cap is the only force that ever destroys data, and it does so in eligibility-priority order.

### The four tiers

| Tier | What it stores | What's recoverable | Storage cost | TTL eligibility |
|---|---|---|---|---|
| **Active** | Current truth | Everything, current state | Full | Forever (until superseded or deleted) |
| **Superseded** | Old version after content change | Everything, via `as_of` and `entity_timeline` | Full row + vec embedding | Eligible to tombstone after 1 year |
| **Tombstoned** | Bare existence (timestamp + id), content stripped | "X existed at T but the content is gone" — entity skeleton + dates only | ~50 bytes per row, no vec | Eligible to hard-delete after 1 year (2 years from supersession) |
| **Hard-deleted** | Nothing | Nothing | 0 | N/A |

The key insight is that **tombstoning preserves the timeline** even after the content is gone. A user asking "was there an entity called X around T2?" can still get a yes/no answer. Only the content is sacrificed, not the existence record.

### Schema additions (deferred)

```sql
-- Already added in §10:
-- ALTER TABLE entities ADD COLUMN superseded_at TEXT NOT NULL DEFAULT '';

-- New in this section (deferred PR):
ALTER TABLE entities ADD COLUMN tombstoned_at TEXT NOT NULL DEFAULT '';
ALTER TABLE entities ADD COLUMN last_accessed_at TEXT NOT NULL DEFAULT '';

ALTER TABLE observations ADD COLUMN tombstoned_at TEXT NOT NULL DEFAULT '';
-- observations don't need last_accessed_at — eviction sweeps via parent entity

ALTER TABLE relations ADD COLUMN tombstoned_at TEXT NOT NULL DEFAULT '';
```

### The observer effect on `last_accessed_at`

The user caught a critical design bug in the original draft: *"the last-accessed timestamp would be updated the moment we read the last-accessed timestamp, wouldn't it?"* Yes — and that would make nothing eligible for eviction, ever. The shield would shield itself.

**Resolution: access discipline + raw-SQL eviction path.**

| Operation | Updates `last_accessed_at`? |
|---|---|
| `read_graph` (no filter, bulk) | **No** — bulk reads aren't intentional access |
| `read_graph(asOf=...)` (historical) | **No** — historical reads are about the past, not the present |
| Eviction sweep | **No** — the sweep is the consumer of this field, never the producer |
| `search_nodes` | **Yes** — targeted search expresses intent |
| `open_nodes` | **Yes** — naming an entity is intent |
| `entity_timeline` | **Yes** — viewing history expresses intent |
| Any write (`add_observations`, `supersede_observations`, etc.) | **Yes** — touching the entity is the strongest signal |

**Critical:** the eviction sweep queries entities via raw SQL through the better-sqlite3 connection directly. It does NOT route through `readGraph` or any other GraphStore method that might trigger update side effects. This is enforced by giving the eviction module its own narrow query helper that *only* SELECTs from the entities table and never writes.

### LRU shield — 6 months

An entity whose `last_accessed_at` is within 6 months is **shielded from eviction at every tier**, regardless of how long it's been superseded or tombstoned. This protects long-tail recall: a project the user returns to after a year of inactivity gets its entire history intact, because the act of returning (via `search_nodes` or `open_nodes`) refreshes the access timestamp on every entity touched and pushes the shield window forward.

The 6-month window is the conservative starting point. It can be tightened later if the cap binds frequently — but never shortened below the longest plausible "I'm coming back to this project" gap. The user's stated pattern includes 1-2 year project breaks; 6 months is a deliberately under-aggressive starting position so we err on the side of recall over storage.

### Size circuit breaker

```typescript
// At server start and every N writes (TBD — likely every 1000 writes):
const dbSize = fs.statSync(dbPath).size;
const cap = 1_000_000_000;  // 1 GB

if (dbSize > cap * 0.9) {
  // Enter eviction mode. Walk eligibility tiers in order until size drops below 0.85 * cap.
  // Each batch is bounded — never block the main loop for more than ~100ms.
  await evictBatch();
}
```

Eviction order (highest priority first):
1. Hard-delete entities/observations/relations in tier `tombstoned` whose `tombstoned_at` is older than 1 year AND whose entity's `last_accessed_at` is older than 6 months.
2. Tombstone entities/observations/relations in tier `superseded` whose `superseded_at` is older than 1 year AND whose entity's `last_accessed_at` is older than 6 months.
3. (No further tiers — we never hard-delete active data on size pressure alone. If active data fills 1 GB, the user gets a hard error and must intervene.)

The 1 GB cap is configurable via `MEMORY_SIZE_CAP_BYTES` env var.

### Why not delete superseded vec rows on supersede (revisited)

§1's vec retention decision becomes self-consistent under §12: superseded vec rows are tombstone candidates after 1 year, and hard-delete candidates after 2 years. So the "monotonic vec table" promise of §1 is bounded by the eviction system here — we don't grow vec storage indefinitely, we just don't delete it *eagerly*. The cost analysis in §1 (~14 MB/year) is the worst case before §12 ships; after §12 ships, the steady-state vec footprint stabilizes around (1 year of supersessions × 1.5 KB) = ~22 MB.

### Goal trace

| Aspect | Goal A (drift) | Goal B (recall) | Goal C (quality) |
|---|:---:|:---:|:---:|
| Prevents unbounded growth eating Claude's available context | ✓ | | ✓ |
| Eligibility-not-trigger preserves long-tail project history | ✓ | ★ | |
| Tombstone tier preserves timeline existence even after content loss | ✓ | ★ | |
| LRU shield protects projects under active or recent use | | ★ | |

## What this does NOT include

- **Full L0-L3 compression pipeline** (mempalace's `layers.py`) — mempalace generates compressed summaries using an AAAK dialect and manages L1 content automatically. Our approach is simpler: observations are manually assigned to layers by the caller (Claude's hooks or user). Automatic compression/summarization is a v2.0 candidate.
- **Auto-entity detection** (mempalace's `entity_detector.py`) — mempalace detects entities from file content. Our server is an MCP tool -- entity creation is the caller's responsibility. Adding auto-detection would conflate the storage layer with the extraction layer.
- **Graph traversal / cross-references** (mempalace's `palace_graph.py`) — mempalace's "tunnels" find where the same concept bridges different projects via BFS walk. Our `open_nodes` retrieves specific entities with their relations, and `search_nodes` does keyword/vector search, but there's no "walk N hops from entity X" tool. At ~200 entities, manual following via `open_nodes` is tractable. v2.0 candidate when entity count reaches 1000+.
- **"What changed since T" query** — a `get_changes_since(timestamp)` tool returning created/superseded/invalidated items since a given moment. Useful for "what happened while I was away?" after a session break. Read-only query against existing columns -- simple implementation, but the plan is full. v2.0 candidate.
- **Entity aliasing / merge** — "mcp-memory-server" and "memory server" may both exist as entities with observations fragmented across them. **§11 Layer 1 (this PR) prevents future mechanical divergence at write time** (whitespace, case, separators). The remaining gaps — semantic divergence catching (§11 Layer 2) and a `merge_entities(source, target)` tool to consolidate already-divergent existing entities — are deferred. Tracked as a GitHub issue alongside §11 Layer 2.
- **Observation provenance** — who/what recorded this observation? A SessionEnd hook, a manual user instruction, or a code review agent? A `source TEXT` column would enable trust weighting (user-stated facts are more reliable than hook-generated ones). v2.0 candidate.
- **Contradiction detection** (mempalace has this as a utility, not integrated into KG operations) — detecting when two active observations state conflicting facts ("test count is 310" + "test count is 343"). Similarity check catches near-duplicates but not semantic contradictions with different wording. Needs semantic comparison, possibly LLM-in-the-loop. v2.0 candidate.
- **Staleness / decay scoring** — observations about "current state" (test counts, version numbers) decay faster than "rules" (preferences, constraints). A `last_verified` timestamp + decay function would let get_summary downrank stale unverified facts. Needs design work on decay models and verification workflow. v2.0 candidate.
- **ChromaDB / external vector DB** — sqlite-vec is sufficient at our scale. If we hit performance walls at 50K+ observations, we'd partition the vec0 table before adding external dependencies.
- **AAAK compression dialect** — mempalace's token-saving format for L1 summaries. Interesting but highly coupled to their specific prompt engineering. Our system uses natural language.
