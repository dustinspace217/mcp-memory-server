- [ ] **Step 2: Update cursor.ts fingerprints — silent-failure firewall**

The cursor fingerprint is what prevents a caller from paginating page 1 with `asOf=T1` and page 2 with `asOf=T2` and silently getting cross-temporal results. We document the constraint and let the fingerprint enforce it.

```typescript
// readGraph cursor binds to the as_of timestamp it was minted with.
// Reusing a cursor with a different as_of will throw InvalidCursorError.
export function readGraphFingerprint(projectId?: string, asOf?: string): string {
  return `readGraph\0${projectId ?? ''}\0${asOf ?? ''}`;
}

export function searchNodesFingerprint(projectId?: string, query?: string, asOf?: string): string {
  return `searchNodes\0${projectId ?? ''}\0${query ?? ''}\0${asOf ?? ''}`;
}
```

> **Why fingerprint, not persist-asOf-in-payload:** Code-reviewer offered both options; silent-failure-hunter and test-analyzer both pushed for fingerprint because persisting asOf in the cursor body would silently swallow caller mistakes (paginating with an inconsistent timestamp would "just work" against the wrong temporal context). Fingerprint surfaces the mistake as a hard error.

- [ ] **Step 3: Add three temporal filter helpers in sqlite-store.ts**

Three helpers, one per table that has both a `created_at` and a `superseded_at` column. They all share the same shape: half-open interval `[created_at, superseded_at)`.

```typescript
// Builds the SQL clause + bound params for "rows active at asOf".
// Half-open interval: created_at <= asOf AND (still active OR superseded strictly after asOf).
// Strict undefined check (NOT !asOf) so that asOf === '' is rejected upstream as a hard error.
private temporalObsFilter(asOf?: string): { clause: string; params: string[] } {
  if (asOf === undefined) return { clause: `o.superseded_at = ''`, params: [] };
  return {
    clause: `o.created_at <= ? AND (o.superseded_at = '' OR o.superseded_at > ?)`,
    params: [asOf, asOf]
  };
}

private temporalRelFilter(asOf?: string): { clause: string; params: string[] } {
  if (asOf === undefined) return { clause: `r.superseded_at = ''`, params: [] };
  return {
    clause: `r.created_at <= ? AND (r.superseded_at = '' OR r.superseded_at > ?)`,
    params: [asOf, asOf]
  };
}

// Entities are temporal once Task 3a lands (soft-delete via superseded_at).
// Note: entities have NO created_at column today — Task 3a does NOT add one;
// instead, "entity exists at asOf" reduces to "soft-delete sentinel hasn't fired yet."
// If/when entity created_at is added, change to created_at <= ? AND (...).
private temporalEntFilter(asOf?: string): { clause: string; params: string[] } {
  if (asOf === undefined) return { clause: `e.superseded_at = ''`, params: [] };
  return {
    clause: `(e.superseded_at = '' OR e.superseded_at > ?)`,
    params: [asOf]
  };
}
```

Thread `asOf` through `buildEntities()` (and apply `temporalEntFilter` to the entity row query around `sqlite-store.ts:1400`), `getConnectedRelations()`, `readGraph()`, `searchNodes()`, and `openNodes()`. The non-as_of path is unchanged: `WHERE superseded_at = ''` continues to be the fast path.

Reject `asOf === ''` at the helper boundary with a clean error: `throw new Error("asOf cannot be empty string — pass undefined for current state or a valid ISO 8601 UTC timestamp")`. This is the third state of the three-state contract (undefined / valid ISO / throw on empty).

- [ ] **Step 4: Vector search at as_of — two-part fix**

```sql
-- via partial unique index on active rows only.
--

-- ACTIVE rows (where superseded_at = '') so historical rows can share names freely.

BEGIN TRANSACTION;

CREATE TABLE entities_new (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  entity_type TEXT NOT NULL,
  project TEXT,
  updated_at TEXT NOT NULL DEFAULT '',
  created_at TEXT NOT NULL DEFAULT '',
  superseded_at TEXT NOT NULL DEFAULT ''
);

INSERT INTO entities_new (id, name, entity_type, project, updated_at, created_at, superseded_at)
SELECT id, name, entity_type, project, updated_at, created_at, ''
FROM entities;

DROP TABLE entities;
ALTER TABLE entities_new RENAME TO entities;

-- Partial unique index: enforces "one active row per name" but allows
-- historical rows to coexist with the active row of the same name.
CREATE UNIQUE INDEX idx_entities_name_active ON entities(name) WHERE superseded_at = '';

CREATE INDEX idx_entities_updated ON entities(updated_at DESC, id DESC);

UPDATE schema_version SET version = 7;

COMMIT;
```

- [ ] **Step 3: Rewrite `deleteEntities()` as soft-delete (atomic three-way update)**

At `sqlite-store.ts:1023-1048`, replace the hard `DELETE FROM entities WHERE name = ?` with a single-transaction soft update that supersedes the entity AND all of its active observations AND all of its active relations at the SAME timestamp:

```typescript
async deleteEntities(entityNames: string[]): Promise<void> {
  if (entityNames.length === 0) return;
  const now = new Date().toISOString();  // single timestamp shared by all three updates

  const tx = this.db.transaction((names: string[]) => {
    const supersedeEntity = this.db.prepare(
      `UPDATE entities SET superseded_at = ? WHERE name = ? AND superseded_at = ''`
    );
    const supersedeObs = this.db.prepare(
      `UPDATE observations SET superseded_at = ?
       WHERE entity_id IN (SELECT id FROM entities WHERE name = ?) AND superseded_at = ''`
    );
    const supersedeRels = this.db.prepare(
      `UPDATE relations SET superseded_at = ?
       WHERE (from_entity = ? OR to_entity = ?) AND superseded_at = ''`
    );

    for (const name of names) {
      supersedeObs.run(now, name);
      supersedeRels.run(now, name, name);
      supersedeEntity.run(now, name);
    }
  });

  tx(entityNames);
  // No syncEmbedding call: vec rows are retained (Task 3 Step 4 Part B).
}
```

`entityTimeline()` queried at exactly `now` shows a clean cut: entity, all its observations, and all its relations all flip to historical at the same instant.

## Post-ship deviation review (2026-04-11)

The shipped code is a **superset** of the plan, not a literal implementation of it. Eleven distinct deviations were surfaced during the post-ship audit, plus five cross-cutting unintended-behavior risks. Each is documented here so a future session reading the plan doesn't have to reverse-engineer *why the code doesn't match the prose above*.

Every deviation traces back to one of three root causes:

1. **Goal A (drift reduction) forced a retroactive migration.** The plan prescribed a tool-boundary-only fix, which would have left every legacy name in the live DB inconsistent forever.
2. **Goal C (quality conversations/code) forced display preservation.** The plan's "display form is NOT preserved" wording would have made Claude read `"dustinspace"` instead of `"Dustin Space"` on every single lookup, indefinitely.
3. **Robustness to legacy state** forced the `PRAGMA foreign_keys = OFF/ON` wrap and the migration's collision detection. Real databases had FK clauses the plan didn't anticipate.

### Deviation 1 — Scope: tool-boundary only → full v7→v8 schema migration

- **What changed:** The plan described normalization wired into `index.ts` tool handlers with no schema change. The ship adds a dedicated `entities.normalized_name` column, rewrites relation endpoints to normalized form, drops `idx_entities_name_active`, and creates `idx_entities_normalized_active`.
- **Why:** A tool-boundary-only fix can't retroactively clean up the ~200 entities already in the live `memory.db`. Legacy surface variants would have persisted indefinitely — exactly the drift Goal A prohibits.
- **Principle:** Goal A beats minimal-scope schema change. If the existing tool (empty normalization on legacy data) doesn't cover the gap, the anti-bloat gate doesn't block the fix — it demands a real one.
- **Impact / risk:** Migration is irreversible without a hand-written downgrade script. If a pre-existing database has two active entities that normalize to the same key, the migration aborts and the server refuses to start until the user manually renames one. Loud-fail by design, but it's a new failure mode for startup.
- **Worth it because:** without this, every existing `dustin-space` / `Dustin Space` / `Dustin_Space` surface variant in the user's live DB would have stayed fragmented forever.

### Deviation 2 — Display form preservation (Option B hybrid)

- **What changed:** Plan: *"The original (display) form is NOT preserved in v1.1."* Ship: `entities.name` preserves the exact form the caller typed; `entities.normalized_name` is the identity key; `readGraph()`/`openNodes()`/`entityTimeline()` translate relation endpoints back to display form via `LEFT JOIN entities ... COALESCE(e.name, r.from_entity)`.
- **Why:** Goal C (quality conversations/code). Claude is the primary consumer of tool output; reading `"dustinspace"` instead of `"Dustin Space"` degrades recall quality on every lookup. The plan conflated *Layer 2* (semantic aliases like "NYC" ↔ "New York City", legitimately deferred) with *surface-form preservation* (a different, cheaper problem).
- **Principle:** Goal hierarchy overrides implementation convention. The anti-bloat gate is a cost/benefit check, not a veto — one `TEXT` column is proportional to the payback.
- **Impact / risk:**
  - **First-write-wins on display form.** Submitting `"Dustin Space"` first, then `"DUSTIN SPACE"`, makes the second a collision — the original capitalization sticks forever and there's no rename primitive.
  - **Leading/trailing whitespace survives in display form.** The normalizer trims *inside itself*, but whatever string the caller passed is what lands in `entities.name`. `"  Dustin Space  "` stores with spaces.
  - **Callers reading `entities.name` no longer get a unique key.** Any future migration or code assuming `name` is unique is wrong — that property moved to `normalized_name`. The FK drop on `relations` (v6→v7) happened for exactly this reason and is a permanent consequence.
- **Worth it because:** the one user that matters for this server (Claude in-session) reads entity names dozens of times per conversation. Sacrificing that for "one fewer column" was a false economy.

### Deviation 3 — File path: `src/utils/normalize-name.ts` → flat root

- **What changed:** Plan: *"Add: `src/utils/normalize-name.ts` (small helper)."* Ship: `/home/dustin/Claude/mcp-memory-server/normalize-name.ts`.
- **Why:** The codebase is flat-root — `sqlite-store.ts`, `jsonl-store.ts`, `embedding.ts`, `cursor.ts`, `types.ts`, `index.ts` all live at the project root. Creating `src/utils/` for one file would have introduced a layout inconsistency the rest of the codebase doesn't have.
- **Principle:** Follow existing project conventions; don't invent hierarchy for a one-file feature.
- **Impact / risk:** None. TypeScript module resolution works identically.
- **Worth it because:** the plan's path was an uncritical carryover from a "proper src/" mental model that doesn't match this codebase. Matching convention costs nothing.

### Deviation 4 — Boundary normalization pattern: inline call → Zod `.refine()`

- **What changed:** Plan: *"normalize the incoming name BEFORE checking the database"* — implying `const normalized = normalizeEntityName(input)` at the top of each tool handler. Ship: a reusable Zod schema `EntityNameInputSchema` (`index.ts:106`) that runs `normalizeEntityName()` inside a `.refine()` predicate, **discards the result**, and only validates that the name doesn't throw. The actual normalization-as-lookup happens inside `SqliteStore` per call.
- **Why:** Option B requires the store layer to know the identity key (for unique-index lookups and relation-endpoint comparison), which means the store must normalize regardless. Doing it twice at two layers would be redundant. The tool boundary's remaining job is *validation* — catching `"---"` or `"   "` before they hit SQL so the caller gets a clean Zod error instead of an opaque error from the bottom of the store.
- **Principle:** Defense-in-depth at the boundary, canonical-identity-computation at the lowest common denominator (the store). Separate responsibilities rather than duplicate logic.
- **Impact / risk:**
  - Normalization is computed twice per call (once in `refine`, once in the store). Idempotent and cheap, but redundant.
  - **Contribution footgun:** anyone adding a new tool that accepts an entity name must remember to use `EntityNameInputSchema` rather than plain `z.string().min(1).max(500)`. If they forget, early validation is lost and the error surfaces deeper in the stack.
- **Worth it because:** the Zod refine pattern composes across every schema that embeds a name, including relation `from`/`to` — a single helper beats sprinkling `normalizeEntityName()` calls across seven handlers.

### Deviation 5 — Deliberate asymmetry: delete/read tools are NOT boundary-validated

- **What changed:** Plan Step 2: apply normalization to *all* of `create_entities`, `delete_entities`, `read_graph`, `open_nodes`, `add_observations`, `supersede_observations`, `entity_timeline`. Ship: `EntityNameInputSchema` is used for **write and observe** paths (`create_entities`, `add_observations`, `supersede_observations`, relation `from`/`to`) but **deliberately not** for `delete_entities`, `open_nodes`, or other read/idempotent-delete paths. See `index.ts:102-104` and `sqlite-store.ts:2267` for the explicit comments.
- **Why:** Idempotent-delete semantics. If `delete_entities(["---"])` raises a Zod validation error, that breaks the contract — callers expect "delete what exists, silently skip what doesn't." Same for `open_nodes`: passing a batch of names, some of which are garbage, should return the ones that exist rather than reject the entire batch.
- **Principle:** Idempotency > uniform boundary validation. The caller contract is a harder constraint than "normalize everywhere."
- **Impact / risk:**
  - **Silent no-op on garbage names.** `delete_entities(["Dustin Space", "---"])` reports one deletion and no errors — the `"---"` gets swallowed. Intended, but debugging-hostile if the caller typo'd a real name as gibberish.
  - **The asymmetry isn't obvious from tool definitions.** A reader skimming `index.ts` sees `EntityNameInputSchema` on some tools and plain `z.string()` on others and may flag it as inconsistency. The JSDoc on `EntityNameInputSchema` explains it, but a reviewer has to read the comment.
- **Worth it because:** breaking the idempotent-delete contract to chase uniform validation would have caused more pain than the silent-skip behavior does.

### Deviation 6 — Relations storage: unspecified → normalized endpoints + JOIN display

- **What changed:** Plan: never addressed how `relations.from_entity`/`to_entity` should be stored (under the plan's "display NOT preserved" model, it didn't matter). Ship: relation endpoints store the **normalized** identity key, not the display form. `readGraph()`/`searchNodes()`/`openNodes()`/`entityTimeline()` all use `LEFT JOIN entities ON entities.normalized_name = relations.from_entity` with `COALESCE(e.name, r.from_entity)` to translate back to display form at read time. The v7→v8 migration rewrites every existing relation.
- **Why:** Option B implies one canonical identity per entity. Relation lookups must hit that canonical identity to work regardless of which surface variant the caller used. If relations stored display form, `createRelation("Dustin-Space", "phase b task 3")` and later lookups of `"dustin-space"` would miss each other.
- **Principle:** One identity key, one storage form, translate at the display layer — keeps the join point explicit and centralized.
- **Impact / risk:**
  - **Raw SQL against `relations.from_entity`/`to_entity` must use the normalized key, not the display form.** This already bit the test suite: `knowledge-graph.test.ts` had a raw SQL query for `'Hub'` that had to be rewritten to `'hub'`. Any future SQL written by a session that hasn't internalized this distinction will be wrong.
  - **`COALESCE(e.name, r.from_entity)` hides dangling relations.** If an entity is ever hard-deleted out from under a relation, the LEFT JOIN falls through and the caller sees the normalized key as the "display name" — namespace mixing. Soft-delete semantics protect against this, but the guard is the caller contract rather than a schema constraint.
  - **Debugging with `sqlite3` or a raw DB browser shows normalized relation endpoints.** Ugly to read at the CLI.
- **Worth it because:** the alternative — storing display-form endpoints and doing normalization at query time on both sides — would have made every relation lookup O(n) in the worst case. The JOIN is cheap and correct.

### Deviation 7 — Test location: `mcp-tools.test.ts` → three files

- **What changed:** Plan Step 3: five test cases in `__tests__/mcp-tools.test.ts`. Ship: tests distributed across three files by responsibility — `__tests__/normalize-name.test.ts` (17 unit tests for the pure function), `__tests__/knowledge-graph.test.ts` SqliteStore-specific `"entity name normalization (Layer 1)"` block (surface-variant behavioral tests against the store), `__tests__/migration-validation.test.ts` (v7→v8 forward + collision-abort). `mcp-tools.test.ts` did grow (35 → 71), but mostly with tool-boundary validation tests.
- **Why:** Test-responsibility colocation. Unit tests for a pure function belong alongside the function. Behavioral tests against the store belong with store tests. Migration tests belong in the migration validation file.
- **Principle:** Tests sit at the layer they test. A migration test in `mcp-tools.test.ts` would be fighting gravity.
- **Impact / risk:** Minimal. Coverage is broader than the plan asked for. The only risk is a reader looking for normalization tests in `mcp-tools.test.ts` because the plan said so — mitigated by this deviation note.
- **Worth it because:** cramming unit, integration, and migration tests into one file would have made that file the dumping ground for any future feature touching entity names.

### Deviation 8 — New field: `SkippedEntity.existingName`

- **What changed:** Plan: no `types.ts` change. Ship: `SkippedEntity` gained an optional `existingName` field so that when a caller submits `"Dustin_Space"` and collides with an existing entity stored under display form `"Dustin Space"`, the collision report can show both forms.
- **Why:** Option B's whole premise is that display form matters. A collision error that only shows the normalized key (`"dustinspace already exists"`) defeats the purpose — the caller has no idea which surface variant already lives in the DB.
- **Principle:** Goal C again — errors are read by Claude in the moment and need to carry enough display info to be actionable.
- **Impact / risk:**
  - **Parameterized test asymmetry.** `knowledge-graph.test.ts` runs its shared behavioral suite against both `JsonlStore` and `SqliteStore`. JsonlStore doesn't track normalized names (v8 is SQLite-only), so it can't populate `existingName`. Collision-report tests now use `toMatchObject()` instead of `toEqual()` because JsonlStore omits the field. Small but permanent maintenance tax on any shared test touching collision reporting.
- **Worth it because:** the collision report was the one place where Option B's promise of display-form readability *had* to reach the caller — without it, the whole hybrid felt pointless on the error path.

### Deviation 9 — Migration collision check with rollback

- **What changed:** Plan: no migration → no migration edge cases. Ship: before creating `idx_entities_normalized_active`, the migration groups active entities by `normalized_name` and counts members. If any group has more than one active row, the migration aborts with a message like `"Migration v7→v8 aborted: collision detected. [...] - ["Dustin Space", "dustin-space"] all normalize to "dustinspace""` and rolls the whole transaction back. `schema_version` stays at 7 and the server fails to start.
- **Why:** You can't create a unique index on a column with duplicates. Silently dropping one colliding row destroys data. Silently renaming one loses reference integrity. The only safe option is loud-fail.
- **Principle:** Data-preserving-before-data-destructive (`feedback_data_preservation_order.md`). Fail loud when automated resolution would lose information.
- **Impact / risk:**
  - **No automatic resolution path.** If the live `memory.db` had such a collision, the server would refuse to start and the user would have to manually open the DB in `sqlite3`, figure out which rows are colliding, and rename one. No CLI tool exists for this.
  - The collision error message only surfaces in stderr as the startup exception — readable but easy to miss in a log scroll.
- **Worth it because:** the alternative (silent data loss) would have violated the foundational data-preservation rule. Loud-fail is the only defensible default here.

### Deviation 10 — `PRAGMA foreign_keys = OFF/ON` wrap on the migration

- **What changed:** Plan: no mention. Ship: the entire v7→v8 migration is wrapped with `this.db.pragma('foreign_keys = OFF')` before the transaction and `'foreign_keys = ON'` after, with both pragma calls placed **outside** the transaction block (SQLite rejects `PRAGMA foreign_keys` inside an open transaction).
- **Why:** Discovered during testing. Legacy databases can arrive at `schema_version = 7` while still carrying FK clauses on `relations.from_entity`/`to_entity` — the v6→v7 migration's "DROP FK" logic only runs on the table-rebuild path, but a database that hit the `CREATE TABLE IF NOT EXISTS` path has the FK baked into the original definition and is never rewritten. Rewriting `relations.from_entity` to normalized form then triggers `"foreign key mismatch"` because the new value doesn't match any `entities.name`. The live project-root `memory.db` hit this exact failure mode.
- **Principle:** Robustness to legacy state. A migration can't assume a clean-slate predecessor because the codebase has multiple paths that produced schema v7.
- **Impact / risk:**
  - **A broken DB with actual orphans would be harder to diagnose if it slipped past the orphan check.** FKs-off means SQLite's own enforcement is temporarily disabled during the window. The orphan check (re-read `relations`, cross-reference against `entities.normalized_name`) is the sole defense. If that check ever had a bug, an orphan would pass silently instead of raising a FK mismatch.
- **Worth it because:** without the pragma wrap, the live `memory.db` and anyone else running on a legacy v7 database would have had their server refuse to start post-upgrade. The explicit orphan check preserves integrity without blocking the migration.

### Deviation 11 — Tool description wording

- **What changed:** Plan suggested: *"Names are normalized: case-insensitive, separators ignored. `Dustin-Space` and `dustin/space` map to the same entity."* Ship: tool descriptions mention normalization but don't adopt the exact wording, and notably don't advertise first-write-wins semantics on display form.
- **Why:** The plan's wording was drafted for "display NOT preserved" semantics. Under Option B the behavior differs — surface variants collapse, but the caller's first-written display form sticks. The plan wording would be slightly misleading under Option B.
- **Principle:** Documentation correctness over literal plan adherence.
- **Impact / risk:** Tool descriptions could be more explicit about first-write-wins semantics. A caller reading the current description might not realize that submitting `"Dustin Space"` after an existing `"dustin-space"` gets rejected rather than "upgrading" the display form. Low-severity doc gap — worth a follow-up edit.
- **Worth it because:** adopting the plan's wording verbatim would have been actively misleading.

## Cross-cutting unintended behaviors (flag list)

These aren't tied to a single deviation, but they emerge from the composition of multiple. Flagging them here so the next maintainer doesn't have to re-derive them.

1. **`search_nodes` LIKE substring search runs on `entities.name` (display form), not `normalized_name`.** If a user created an entity as `"Dustin Space"` and later does `search_nodes({ query: "dustin-space" })`, the LIKE pattern `%dustin-space%` doesn't substring-match `"Dustin Space"` and the entity won't be returned. Vector search *may* rescue it semantically if the embedding model is loaded, but LIKE substring is blind to this. **This is a genuine Goal C regression that Option B introduced and nothing in this ship mitigates.** Suggested follow-up: normalize the query and LIKE-search `normalized_name` as well, or LIKE-search both columns.

2. **Double normalization per write.** Every `createEntities`/`addObservations`/`supersedeObservations` call normalizes once in Zod `.refine()` (to validate) and once in `SqliteStore` (to compute the identity key). Idempotent and cheap, but latent "why is this function called twice?" for future perf work.

3. **Migration audit watermarks drift.** The live `memory.db` migrated automatically, but `/tmp/claude/last-memory-audit.timestamp` (used by the freshness system) wasn't touched. Tier-1 freshness scans still flag entities correctly by file mtime, but the *schema-change* event isn't recorded anywhere the freshness system can see.

- [x] **Step 1: Implement the normalizer**

```typescript
// src/utils/normalize-name.ts
//
// Normalizes an entity name for storage and lookup. Layer 1: deterministic,
// reversible-where-practical, no semantic understanding.
//
// 1. Trim outer whitespace
// 2. Unicode NFC normalize (canonical compose) — guarantees ä == ä
// 3. Case-fold to lowercase (Unicode-aware)
// 4. Strip ALL separators: hyphens, underscores, slashes, dots, colons, spaces
// 5. Reject empty result
//
// Why NFC over NFKC: NFC preserves visual identity. NFKC would collapse
// "ﬁ" → "fi" and "Ⅸ" → "IX", which destroys information when the user
// genuinely meant the typographic ligature or roman numeral form.
//
// Caveat: Turkish dotted-I (İ → i̇, dotless ı) — case folding here uses
// Default (en-US) locale. Acceptable for English-dominant projects; if a
// future user is Turkish, document this in the README.
export function normalizeEntityName(input: string): string {
  const trimmed = input.trim();
  if (trimmed.length === 0) {
    throw new Error('Entity name cannot be empty');
  }
  const folded = trimmed.normalize('NFC').toLowerCase();
  const stripped = folded.replace(/[\s\-_/.\\:]+/g, '');
  if (stripped.length === 0) {
    throw new Error('Entity name has no content after normalization');
  }
  return stripped;
}
```

- [ ] **Step 1: Create audit.ts**

```typescript
import * as fs from 'node:fs/promises';

/**
 * Append-only JSONL audit logger for knowledge graph mutations.
 * Each log entry records the operation name, a summary of the input
 * (not full content -- just names and counts), and the outcome.
 *
 * Fire-and-forget from MCP handlers -- failures are logged to stderr
 * but never propagate to callers.
 */
export class AuditLogger {
  // Path to the .audit.jsonl file, derived from the main database path
  private logPath: string;
  // Whether logging is enabled (disabled by MEMORY_AUDIT_LOG=off)
  private enabled: boolean;

  constructor(dbPath: string, enabled: boolean = true) {
    this.logPath = dbPath + '.audit.jsonl';
    this.enabled = enabled;
  }

  /**
   * Appends a single audit entry to the log file.
   * Creates the file on first write. Safe for single-writer (this process).
   *
   * @param op - The operation name (e.g., 'create_entities', 'add_observations')
   * @param input - Summary of the operation input (names, counts -- not full content)
   * @param result - Summary of the outcome (counts, success/failure)
   */
  async log(
    op: string,
    input: Record<string, unknown>,
    result: Record<string, unknown>
  ): Promise<void> {
    if (!this.enabled) return;
    try {
      const entry = JSON.stringify({
        ts: new Date().toISOString(),
        op,
        input,
        result,
      });
      await fs.appendFile(this.logPath, entry + '\n');
    } catch (err) {
      // Audit logging must never fail the operation -- log to stderr and continue
      console.error(`Audit log write failed: ${err}`);
    }
  }
}
```

```typescript
// Token budget enforcement: L0 ~100 tokens (~400 chars), L1 ~800 tokens (~3200 chars)
const L0_CHAR_BUDGET = 400;
const L1_CHAR_BUDGET = 3200;

let l0CharCount = 0;
const warnings: string[] = [];

// L0: include all but warn if over budget (never truncate L0 -- it's core rules)
for (const obs of l0Results) {
  l0CharCount += obs.content.length + obs.entityName.length + 5;
}
if (l0CharCount > L0_CHAR_BUDGET) {
  warnings.push(`L0 exceeds 100-token budget (est: ${Math.ceil(l0CharCount / 4)} tokens)`);
}

// L1: include until budget exhausted, sorted by importance DESC
let l1CharCount = 0;
const l1Included: typeof l1Results = [];
for (const obs of l1Results) {
  const cost = obs.content.length + obs.entityName.length + 5;
  if (l1CharCount + cost > L1_CHAR_BUDGET) break;
  l1Included.push(obs);
  l1CharCount += cost;
}

const totalChars = l0CharCount + l1CharCount;
return {
  L0: l0Results,
  L1: l1Included,
  tokenEstimate: Math.ceil(totalChars / 4),
  warnings
};
```

```typescript
setObservationMetadata(updates: SetObservationMetadataInput[]): Promise<number> {
  let totalUpdated = 0;
  const txn = this.db.transaction(() => {
    for (const u of updates) {
      // Look up entity
      const entity = this.db.prepare('SELECT id FROM entities WHERE name = ?').get(u.entityName);
      if (!entity) throw new Error(`Entity not found: ${u.entityName}`);

      // Build SET clause dynamically based on which fields are provided
      const sets: string[] = [];
      const params: unknown[] = [];
      if (u.importance !== undefined) { sets.push('importance = ?'); params.push(u.importance); }
      if ('contextLayer' in u) { sets.push('context_layer = ?'); params.push(u.contextLayer); }
      if ('memoryType' in u) { sets.push('memory_type = ?'); params.push(u.memoryType); }
      if (sets.length === 0) continue;

      params.push(entity.id, u.content);
      const result = this.db.prepare(
        `UPDATE observations SET ${sets.join(', ')} WHERE entity_id = ? AND content = ? AND superseded_at = ''`
      ).run(...params);
      totalUpdated += result.changes;

      // Update entity's updated_at if any observation was modified
      if (result.changes > 0) {
        this.db.prepare('UPDATE entities SET updated_at = ? WHERE id = ?')
          .run(new Date().toISOString(), entity.id);
      }
    }
  });
  txn();
  return Promise.resolve(totalUpdated);
}
```

## Future / v2.0 candidates

| Feature | What it solves | Why deferred |
|---------|---------------|-------------|
| **"What changed since T"** | After session break: "what happened while I was away?" A `get_changes_since(timestamp)` returning created/superseded/invalidated items. | Read-only query against existing columns. Simple, but plan is full. |
| **Entity aliasing / merge** | "mcp-memory-server" and "memory server" fragment observations across duplicates. `merge_entities(source, target)` consolidates. | Needs schema design (alias table or redirect), cascading relation updates. |
| **Observation provenance** | Who/what recorded this? SessionEnd hook vs. user instruction vs. agent. `source TEXT` column enables trust weighting. | Small schema change, but needs taxonomy design for source values. |
| **Contradiction detection** | Two active observations state conflicting facts. Similarity check catches near-duplicates but not semantic contradictions. | Needs semantic comparison, possibly LLM-in-the-loop. Mempalace has this but hasn't integrated it either. |
| **Staleness / decay scoring** | "Current state" observations (test counts, versions) decay faster than "rules" (preferences). `last_verified` timestamp + decay function. | Needs decay model design and verification workflow. |

| **Auto-entity detection** (mempalace's `entity_detector.py`) | Detect people/projects from text automatically. | Conflates storage layer with extraction layer. Entity creation is the caller's responsibility. |
